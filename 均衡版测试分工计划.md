# 均衡版测试分工计划

版本：2026-04-01  
适用目录：`D:\C++\his1-main\his1-main\xitong`

## 1. 分工原则

这份计划按“当前代码可执行”和“每个人难度相当”重新拆分。

平衡原则：

- 每个人都包含 1 组基础功能测试
- 每个人都包含 1 组边界/异常测试
- 每个人都包含 1 组跨模块联动测试
- 每个人都包含 1 组结果核验测试

本轮不纳入正式分工的项目：

- AI 智能推荐
- 急诊大屏标红置顶
- ASCII 艺术报告单
- 排班防分身锁

原因：这些点要么是当前代码未实现，要么需要调整口径，不适合作为本轮正式 bug 指标。

## 2. 统一测试准备

开始前统一完成以下准备：

1. 备份 `data` 目录。
2. 确认程序可正常启动：`his_system.exe`
3. 准备 3 到 5 个患者编号。
4. 准备多个医生工号，至少覆盖两个科室。
5. 如需测试急诊/门诊隔离，先备份并修改 `data\schedules.txt`。

## 3. 组员 A：患者入口与挂号防线

角色定位：前台用户链路负责人

难度评估：中高  
特点：数据准备多、边界条件多、需要频繁切换患者端与医生端核对结果。

### A-1 输入防呆与金额极值

测试范围：

- 充值金额
- 菜单选项
- 医生工号输入

测试数据：

- `abc`
- 直接回车
- `-50`
- `200000`
- `12x`
- `8.5x`

预期结果：

- 非法整数被拦截
- 非法金额被拦截
- 负数金额被拦截
- 超过 100000 的金额被拦截
- 程序不崩溃、不乱跳菜单

### A-2 挂号限流四件套

测试范围：

- 全院每日 500 号
- 单医生每日 20 号
- 单患者每日 5 号
- 同患者同科室每日 1 号

执行要求：

- 这四条必须分开设计数据，不要互相抢先触发。
- 优先用多个患者测试“单医生 20 号”。
- 优先用多个医生测试“单患者 5 号”。

预期结果：

- 每条拦截线都能独立验证
- 提示语与拦截条件一致
- 不出现误拦截或漏拦截

### A-3 满号医生与不存在医生处理

测试范围：

- 输入已满号医生
- 输入不存在医生

预期结果：

- 已满号医生会触发同科室候选医生列表
- 不存在医生会提示“未找到该医生”
- 推荐结果只要求正确，不要求 AI 排序能力

### A-4 排队状态与预计时间

测试范围：

- 已缴费挂号进入等待队列
- 前方人数计算
- 预计等待时间计算

执行方式：

1. 为同一医生准备多条已缴费挂号。
2. 让目标患者排在后面。
3. 在患者端查看排队状态。

预期结果：

- 前方人数正确
- 急诊按每人 5 分钟估算
- 门诊按每人 12 分钟估算

### A-5 患者缴费闭环

测试范围：

- 挂号后缴费
- 电子票据生成
- 余额扣减正确

预期结果：

- 支付后记录状态变为已缴费
- 患者余额正确减少
- 票据信息完整

## 4. 组员 B：医生工作流与临床联动

角色定位：接诊流程与医疗规则负责人

难度评估：中高  
特点：涉及场景切换、接诊状态机、多个医生功能串联，业务判断较多。

### B-1 急门诊隔离

测试范围：

- 急诊医生登录
- 门诊医生登录
- 候诊队列隔离

执行方式：

1. 修改 `data\schedules.txt`。
2. 分别准备急诊和门诊挂号数据。
3. 让急诊医生、门诊医生分别登录。

预期结果：

- 急诊医生只看到急诊患者
- 门诊医生只看到普通门诊患者
- 不出现混队

### B-2 叫号与接诊状态流转

测试范围：

- 查看候诊队列
- 呼叫下一位患者
- 结束当前接诊
- 未叫号前禁止直接开单

预期结果：

- 当前已有患者时不能重复叫号
- 未叫号时不能直接开处方、检查、住院、手术
- 结束接诊后状态恢复为空

### B-3 处方与发药闭环

测试范围：

- 开立处方
- 处方缴费
- 药房发药
- 库存扣减

建议覆盖：

- 正常发药
- 未缴费处方禁止发药
- 已发药记录禁止重复发药
- 库存不足时禁止发药

预期结果：

- 发药条件判断准确
- 药品库存同步减少
- 无重复发药漏洞

### B-4 检查单闭环与报告排版

测试范围：

- 开立检查单
- 患者缴费
- 医生生成检查报告
- 报告文本完整性

预期结果：

- 未缴费检查单不能生成报告
- 缴费后可生成报告
- 再次生成时会提示已完成
- 报告字段完整，不乱行

### B-5 手术室并发防撞锁

测试范围：

- 同患者同时进两台手术
- 同主刀医生同时开两台手术

预期结果：

- 两类冲突都能被拦截
- 提示信息明确
- 不会出现手术室双占用

### B-6 医生日统计

测试范围：

- 今日接诊量
- 累计收入

预期结果：

- 只统计已缴费且已叫号或已完成的挂号
- 不把未缴费挂号误记入接诊量

## 5. 组员 C：住院、财务与后台落盘

角色定位：后台稳定性与财务安全负责人

难度评估：中高  
特点：覆盖床位、押金、退款、手术室释放、文件落盘，验证深度较大。

### C-1 住院收治押金规则

测试范围：

- 押金是否为 100 的整数倍
- 押金是否满足 `200 x 天数`
- 非法押金是否拦截

建议数据：

- 住院 3 天，分别输入 `500`、`550`、`600`

预期结果：

- `500` 拦截
- `550` 拦截
- `600` 通过

### C-2 合法收治与床位分配

测试范围：

- 合法收治
- 转入住院列表
- 分配空床
- 床位占用更新

预期结果：

- 合法条件下不再误报“无床位”
- 患者能成功转入住院
- 床位和患者信息同步更新

### C-3 ICU 熔断与床位释放

测试范围：

- ICU 有床时正常转入
- ICU 满床时触发告警
- 成功转 ICU 后原床位释放

预期结果：

- 告警只在满床时出现
- 有空床时不误拦截
- 原床位患者绑定被清空

### C-4 财务报表与押金剔除

测试范围：

- 完成住院收治后查看管理端财务报表
- 核对营业额是否包含押金

预期结果：

- 财务报表不应把住院押金计入营收
- 正常业务收费仍应被统计

### C-5 冲红退款防吞钱

测试范围：

- 已缴费普通记录作废
- 患者余额原路退回

执行方式：

1. 患者先挂号并完成缴费。
2. 管理端作废该记录。
3. 回患者端核对余额。

预期结果：

- 退款金额正确
- 记录状态变为已作废
- 不出现重复退款或未退款

### C-6 手术室释放与持久化恢复

测试范围：

- 手术结束后患者移出
- 完成消杀后手术室释放
- 安全退出后文件落盘
- 重启后数据恢复

执行方式：

1. 先安排一台手术。
2. 管理端结束手术并转入消杀。
3. 再释放手术室。
4. 主菜单输入 `0` 安全退出。
5. 检查 `data` 目录文件。
6. 重启程序复核数据。

预期结果：

- 手术室状态流转正确
- 患者与医生字段能清空
- 退出后数据写回成功
- 重启后状态保持一致

## 6. 工作量平衡说明

三个人现在的难度大致平衡如下：

- 组员 A：偏数据量与边界组合，工作繁琐度高
- 组员 B：偏业务判断与流程串联，逻辑复杂度高
- 组员 C：偏后台核账与状态恢复，核验深度高

整体上：

- A 的“准备量”最多
- B 的“逻辑判断量”最多
- C 的“核对深度”最多

三者强度接近，适合并行推进。

## 7. 推荐执行顺序

推荐团队按以下顺序并行：

1. A 先铺底数据与挂号数据
2. B 基于 A 的数据做医生流程测试
3. C 在 B 做完住院、手术后接着核账和恢复测试

如果完全并行，建议：

- A 主要使用患者端
- B 主要使用医生端
- C 主要使用管理端和 `data` 文件复核
